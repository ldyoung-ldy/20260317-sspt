# Step 4 作品提交手工验收脚本

> 用途：在真实数据库与浏览器登录态下，验证 Step 4 的核心业务链路。

## 前置条件

1. 本地环境已配置 `.env.sample` 所需变量，尤其是 `DATABASE_URL`、`AUTH_SECRET` 与 OAuth provider。
2. 至少存在一个已发布赛事，且当前时间落在该赛事 `submissionStart <= now < submissionEnd`。
3. 准备两个账号：
   - 普通用户 A：报名已被管理员接受并已在“我的报名”点击确认参赛
   - 管理员账号 B：可访问 `/admin`

## 验收步骤

### 场景 1：confirmed 用户首次保存草稿

1. 使用普通用户 A 登录。
2. 打开某个已发布赛事详情页 `/events/[slug]`。
3. 确认页面右侧出现“提交作品”入口。
4. 进入 `/events/[slug]/submit`。
5. 填写作品名称、描述、至少一个链接；如果赛事配置了赛道，则选择赛道。
6. 点击“保存草稿”。

预期结果：

- 页面刷新后出现橙色“草稿” Badge。
- “我的作品”页面出现新卡片。
- 卡片显示最近更新时间、赛事名称、赛道和赛题。

### 场景 2：草稿提交为终稿

1. 继续停留在作品提交页，补全需要的信息。
2. 点击“提交终稿”。

预期结果：

- 页面刷新后出现绿色“终稿” Badge。
- “我的作品”页面对应卡片状态同步变为终稿。

### 场景 3：终稿截止前继续修改

1. 在提交窗口仍未关闭的前提下，重新打开 `/events/[slug]/submit`。
2. 修改描述或链接。
3. 点击“更新终稿”。

预期结果：

- 页面刷新后仍保持绿色“终稿” Badge。
- 更新时间发生变化。
- 后台作品管理页看到最新内容。

### 场景 4：非 confirmed 用户拦截

1. 使用未确认参赛或待审核用户登录。
2. 打开 `/events/[slug]/submit`。

预期结果：

- 页面不展示表单。
- 页面显示“需要先获得报名确认才能提交作品”。

### 场景 5：后台查看与导出

1. 使用管理员账号 B 登录。
2. 打开 `/admin/events/[id]/projects`。
3. 检查顶部统计、列表行、详情展开、筛选与 CSV 导出。

预期结果：

- 列表中出现用户 A 的作品。
- “查看详情”可以展开描述、链接、赛题和提交人信息。
- 应用筛选后导出的 CSV 只包含当前筛选结果。
